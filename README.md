# Infrastructure Engineer Portfolio

未経験からインフラエンジニアを目指し、Linux / Nginx / systemd / SSH / ログ確認 / 監視スクリプトを中心に学習しています。

単にコマンドを実行するだけではなく、  
「何が起きているのかを確認する → 原因候補を立てる → ログやコマンドで確認する → 解決する」  
という流れを意識して、実践した内容をGitHubにまとめています。

## 現在取り組んでいること

現在は、Linuxサーバ上で以下のような実践を行っています。

- Nginxを使ったWebサーバ構築
- Python http.serverをupstreamにしたリバースプロキシ構成
- systemdによるサービス管理
- SSHによるリモート操作
- access.log / error.log / journalctl を使ったログ確認
- ss / curl / nc を使った通信確認
- シェルスクリプトによる監視・自動復旧

## Portfolio Repositories

### nginx-troubleshooting-lab

Nginx / systemd / Python http.server を使い、Webサービス障害を再現・切り分け・解決した記録です。

502 / 404 / 403 エラーについて、ログやコマンドを使いながら、Nginx側の問題か、upstream側の問題か、Linux権限の問題かを分けて確認しました。

主に扱った内容は以下です。

- 502 Bad Gateway
  - Pythonアプリ停止
  - proxy_passのポート不一致
  - systemdのExecStartミス
  - WorkingDirectoryミス
  - bind先IPの不一致

- 404 Not Found
  - Python http.server側のindex.html不足
  - alias設定ミス
  - root設定ミス

- 403 Forbidden
  - ファイル権限不足
  - deny all によるアクセス拒否
  - directory index is forbidden
  - indexディレクティブの指定ミス

障害ごとに、症状・期待される状態・実際の状態・原因候補・確認コマンド・切り分け・原因・解決・学びを整理しています。

### linux-monitoring-lab

Linuxサーバの状態確認、ログ確認、監視スクリプト、自動復旧をまとめたリポジトリです。

サービスが動いているかだけでなく、ポートがLISTENしているか、HTTP応答が返るか、ログに異常が出ていないかを組み合わせて確認しました。

主に扱った内容は以下です。

- nginxの起動状態確認
- 80番ポートのLISTEN確認
- HTTPレスポンス確認
- nginx停止時の自動復旧
- access.logからの404確認
- IPごとのアクセス集計
- cronによる定期実行
- 監視結果のログ出力

単に「サービスがactiveか」だけで判断せず、サービス・ポート・HTTP応答・ログを分けて確認することを意識しました。

### ssh-remote-operation-lab

SSHを使ったリモート操作、接続確認、ファイル転送、鍵認証の理解をまとめたリポジトリです。

SSH接続ができない場合に、クライアント側・ネットワーク経路・サーバ側のどこで問題が起きているかを分けて確認する練習をしました。

主に扱った内容は以下です。

- SSH接続
- 公開鍵認証
- known_hosts / authorized_keys の理解
- scpによるファイル転送
- ssh -vvv による接続調査
- nc によるポート疎通確認
- ss による待ち受け確認
- journalctl によるSSHログ確認

リモート接続では、「接続できない」という結果だけでなく、どこまで通信が届いているかを段階的に確認することを意識しました。

## 学習で意識していること

学習では、コマンドを覚えるだけでなく、以下の流れを大切にしています。

1. まず症状を確認する
2. 期待される状態と実際の状態を分ける
3. 原因候補を立てる
4. ログやコマンドで確認する
5. 原因を特定する
6. 解決後に再確認する
7. 学んだことを言語化する

実務では、原因が最初から分かっていることは少ないと思うので、  
ログを見て、仮説を立てて、確認しながら進める力を身につけたいと考えています。
